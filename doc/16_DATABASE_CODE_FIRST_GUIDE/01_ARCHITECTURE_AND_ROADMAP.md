# Database Code First Architecture And Roadmap

## 1. Purpose

This document defines the database development roadmap for the `lms` backend using C# entities first and SQL Server database generation through EF Core migrations.

The goal is to make the database easy to maintain, migration-friendly, and aligned with the Clean Architecture used by the solution.

## 2. Architecture Overview

The solution structure for database development is:

```txt
lms.Domain
  Entity, enum, domain rule

lms.Application
  Use case, DTO, validation, service contract

lms.Persistence
  EF Core DbContext, entity configuration, repository, migration

lms.Api
  Composition root, dependency injection, appsettings.json

SQL Server
  Database generated by EF Core migration
```

Runtime flow:

```txt
lms.WebMvc
  -> HTTP/AJAX
lms.Api
  -> Application service
lms.Persistence
  -> EF Core
SQL Server
```

`lms.WebMvc` must not reference `lms.Domain`, `lms.Application`, `lms.Persistence`, or `lms.Infrastructure`.

## 3. Database Development Direction

Use Code First:

```txt
Design entity in C#
  -> Configure table and columns with EF Core Fluent API
  -> Add DbSet to LmsDbContext
  -> Create migration
  -> Review migration file
  -> Apply migration
  -> SQL Server database is created or updated
```

Do not use:

- Database First
- Scaffold-DbContext
- Manually creating database tables first
- Manual database creation before migration

The initial database should be created by:

```bash
dotnet ef database update
```

## 4. Connection String Rule

The connection string must be read from:

```txt
src/lms/lms.Api/appsettings.json
```

Current expected key:

```json
{
  "ConnectionStrings": {
    "LmsDb": "Data Source=HuyKhoiPad\\SQLEXPRESS;Initial Catalog=lms;Integrated Security=True;Trust Server Certificate=True"
  }
}
```

Rules:

- Do not hardcode connection strings in C#.
- `lms.Api` is the startup project for migration.
- `lms.Persistence` is the migration project.
- Environment-specific values can be overridden in `appsettings.Development.json` or user secrets later.

## 5. Core Database Rules

### 5.1 No Foreign Key Constraint

The SQL Server database must not contain hard foreign key constraints.

Logical relationship example:

```txt
ExamAttempts.UserId -> Users.Id
ExamAttempts.ExamId -> Exams.Id
AttemptAnswers.AttemptId -> ExamAttempts.Id
AnswerOptions.QuestionId -> Questions.Id
```

These relationships are used by query and service logic only. SQL Server must not enforce them.

### 5.2 Application-Level Relationship Validation

Before writing data that references another record, the Application/Service layer must validate existence and business permission.

Examples:

- Before assigning a course, check `UserId` or `GroupId` exists.
- Before starting an exam attempt, check `ExamId` exists and is published.
- Before saving an answer, check `AttemptId` exists and belongs to the current user.
- Before adding an answer option, check `QuestionId` exists.

### 5.3 Primary Key

All normal business tables use:

```txt
Id int identity
```

Do not use `Guid` as the default primary key type.

Use string code or token only for public/external values:

- `CertificateCode`
- `RefreshToken`
- `PublicCode`
- `FileKey`

### 5.4 Audit Fields

All business tables should inherit common audit fields:

```txt
CreatedDate
UpdateDate
CreatedBy
UpdatedBy
IsDelete
```

Use exactly `IsDelete`, not `IsDeleted`, to stay aligned with the database blueprint.

- `AttemptQuestionSnapshots`

### 7.9 Scoring And Results

Entities:

- `ExamResult`
- `ExamResultDetail`

Tables:

- `ExamResults`
- `ExamResultDetails`

`ExamResultDetails` can be added when detailed review, question analytics, or reporting needs become stable.

### 7.10 Certificates

Entities:

- `Certificate`
- `CertificateFile`

Tables:

- `Certificates`
- `CertificateFiles`

`CertificateFiles` is optional if file metadata is centralized in a `Files` table.

### 7.11 Reporting

MVP should not create dedicated report tables.

Reports are generated from:

- `Users`
- `Courses`
- `LearningProgress`
- `Exams`
- `ExamAttempts`
- `ExamResults`
- `Certificates`

Future entities:

- `ReportExportJob`
- `ReportSnapshot`

Add these only when async export or historical report snapshots are required.

### 7.12 Audit Logging

Entity:

- `AuditLog`

Table:

- `AuditLogs`

Audit logs are append-only. Application logic must not update or delete audit logs.

### 7.13 File Management

Entities:

- `FileRecord`

Tables:

- `FileRecords`

Use for:

- Uploaded learning material files
- Certificate PDFs
- Report exports
- Image/media references if needed later

## 10. Key Risks

### EF Core accidentally generates FK constraints

Prevention:

- Do not define navigation properties.
- Do not use `HasOne`, `HasMany`, `WithOne`, `WithMany`, `HasForeignKey`.
- Review every migration file before applying.

### Logical references become invalid

Prevention:

- Validate related records in Application services.
- Add integration tests for important workflows.
- Add indexes on logical reference columns.

### Exam history changes after admin edits questions

Prevention:

- Create attempt question and answer snapshots at exam start.
- Score based on snapshots, not current question bank records.

### Report queries become slow

Prevention:

- Start with normalized transaction tables.
- Add indexes by query pattern.
- Add report snapshot/export job tables only when needed.
